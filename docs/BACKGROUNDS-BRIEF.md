# Бриф для дизайнера — анимированные фоны Private Speed Dial (живые сцены + случайные события)

## Роль (задание агенту)
Ты — **senior motion / generative-UI дизайнер**, специализирующийся на **процедурных фонах с минимальной
нагрузкой** (canvas/CSS, без растровых ассетов и без сборки). Профиль: вектор-арт, параллакс, физика частиц
«на глаз», цвет и композиция «киношных» сцен, плюс инженерная дисциплина по бюджету кадра (rAF, DPR, GPU).
Ты умеешь переводить настроение сцены в **точные параметры** (hex-стопы, число и поведение сущностей,
тайминги событий), которые программист реализует один-в-один — без додумывания.

**Что ты делаешь:** по каждой сцене из §2 выдаёшь готовую спецификацию в формате §4 (CSS-слои или canvas-
спека), укладываясь в бюджет §1; описываешь случайные события (§3) и, для городских сцен, общий слой
погоды/времени/сезона (§3.5). **Чего НЕ делаешь:** не предлагаешь бинарные ассеты, удалённые ресурсы, копи-
райтные обои/бренды, тяжёлый полноэкранный blur/фильтры в движении и «фотореализм». Любой компромисс
производительности — явно отмечаешь и предлагаешь дешёвую альтернативу.

**Критерий «готово»:** спека настолько конкретна, что её можно закодить без вопросов и она проходит §6.

---

Цель: проработать набор **фонов-сцен** для new-tab — статичных и анимированных, включая «живые» сцены со
**случайными событиями**. Ключевое ограничение — **минимальная нагрузка на браузер**. Это промт/бриф:
по каждой сцене нужны конкретные параметры (цвета/слои или спецификация canvas-сцены), пригодные к
реализации без дизайнерских бинарников.

## 0. Технический конверт (обязательно соблюдать)
- **Без сборки (no build):** ваниль JS + CSS, файлы из `src/` отгружаются как есть. Никаких npm-ассетов.
- **CSP / privacy:** только локальные ресурсы, без удалённых картинок/шрифтов/CDN. Расширение «Private».
- **Кросс-браузер:** Chrome / Edge / Firefox / Safari.
- **Без копирайта:** реальные обои (Windows Bliss/Vista, кадры GTA, бренды авто и т.п.) использовать
  НЕЛЬЗЯ — только **оригинальная** стилизация «в духе».
- **Векторно, а не растром:** сцены строим CSS-градиентами и canvas-вектором (пути/фигуры). Больших
  bitmap-картинок избегаем (вес, копирайт, GPU-память).

## 1. Бюджет производительности (главное требование)
Делим фоны на 2 класса:

**A. Статичные сцены — НУЛЕВАЯ runtime-стоимость.** Реализуются как **слои CSS-градиентов**
(`linear/radial/conic`, несколько слоёв = `background-image` через запятую). Идеал для «луг», «осень»,
«аврора». Дизайнер даёт **точные стопы и слои** (см. формат §4). Пример уже в коде: пресет `meadow` =
sun-glow radial + green-hill radial + sky→grass linear.

**B. Анимированные сцены — на canvas, со строгим бюджетом:**
- **Один** `<canvas>` на весь фон (уже есть `#bg-fx`), `requestAnimationFrame`, **пауза при `document.hidden`**
  и **полная остановка**, когда сцена не выбрана.
- Цель: **≤ ~3% CPU** на типичном ноутбуке, стабильные 60fps (или осознанно 30fps для «ленивых» сцен).
- **Мало сущностей:** десятки, не тысячи (рыбки: 6–14, машины города: 8–16). Масштабировать число от
  ширины экрана.
- **GPU-friendly:** анимировать дешёвое (позиции, простые пути). НЕ анимировать `blur`/тяжёлые тени на
  больших областях; не использовать полноэкранный `filter: blur` в движении; не перерисовывать
  «фотореализм».
- **Учитывать `prefers-reduced-motion`** — рисовать один статичный кадр без движения.
- DPR ограничить (≤2), не рисовать в 4K без необходимости.
- Никаких таймеров чаще rAF; не плодить listener'ы при ре-рендере (один раз).

### 3.5 Погода / время суток / сезон (общая система, настраиваемая)
Единый **дешёвый** слой, переиспользуемый всеми «городскими»/сценовыми фонами и настраиваемый
пользователем в настройках:
- **Время суток:** день / закат / ночь (или «авто» по системному времени) — меняет палитру неба и тинт.
- **Сезон:** весна / лето / осень / зима — палитра/акценты (листва, снег на крышах и т.п.).
- **Погода:** ясно / дождь / снег / туман — лёгкий частичный слой на ТОМ ЖЕ canvas:
  - дождь = короткие наклонные штрихи (line), снег = редкие точки с дрейфом, туман = полупрозрачный
    градиент-оверлей (без анимации/blur).
- Бюджет: частиц немного (десятки, не тысячи), один rAF-цикл, **пауза при `hidden`**, **reduced-motion →
  без частиц** (статичный тинт). Масштаб частиц от ширины экрана.
- Дизайнер задаёт: палитры для каждой комбинации (время×сезон), параметры частиц (кол-во, скорость,
  угол/размер) для дождя/снега/тумана, и как это маппится на настройки (перечень опций).

## 3. Система случайных событий (нужно описать для КАЖДОЙ живой сцены)
Сделать сцены «живыми»: помимо постоянного фонового движения — **редкие случайные события**, дешёвые по
ресурсам. Для каждого события дизайнер задаёт:
- **триггер/частота** (например, в среднем раз в 20–60 c; экспоненциальный интервал, не таймер-спам);
- **длительность** и **число сущностей** (1–2 за раз, не залп);
- **что именно рисуется** (вектор-описание/движение);
- **fallback при reduced-motion** (обычно — событие не запускается).

Примеры событий по сценам (расширить):
- **Аквариум:** проплывает крупная рыба/стая; пузырьковый «фонтан»; краб проходит по дну; редкая медуза.
- **Луг/осень:** пролёт птицы/стаи; одинокое облако; падающий лист (осень); бабочка.
- **Город будущего:** **такси** проносится ближней полосой; **полицейская погоня** (2 машины + мигалка);
  пролёт **дирижабля/дрона** с неоновым баннером; вспышка-«реклама» на небоскрёбе; «пробка» из машин на
  дальней полосе; редкий «инцидент» (мигающий перекрёсток) — GTA-настроение, но абстрактно и дёшево.

Технически: один планировщик событий на сцену, событие — это временная сущность в том же rAF-цикле
(никаких отдельных интервалов). Глобальный кап: не более N одновременных событийных сущностей.

## 4. Формат сдачи (что прислать на каждую сцену)
**Для статичных (CSS):** перечень слоёв сверху вниз с типом и стопами, маппящимися на `gradientCss`:
```
scene "autumn":
  layer1 radial-gradient(<pos>, <stops>)   // напр. тёплое солнце
  layer2 linear-gradient(<angle>, <stops>) // листва/небо
```
+ короткое обоснование настроения.

**Для анимированных (canvas):**
- палитра (hex), число сущностей по брейкпоинтам ширины;
- описание каждой сущности: форма (вектор-примитивы), размер, скорость, параллакс-слой, цикл анимации
  (что колеблется и с какой частотой);
- список **случайных событий** по §3 (частота, длительность, что рисуется, кап);
- ожидаемая нагрузка (оценочно: сущностей × сложность) — подтвердить, что укладывается в §1.

## 6. Acceptance / критерии приёмки
- Статичные сцены: 0% CPU в простое, выглядят как сцена (не «плоский градиент»).
- Анимированные: ≤~3% CPU, 60/30 fps, пауза в фоне, корректный reduced-motion, нет утечек listener'ов.
- Случайные события: редкие, не нагружают, не повторяются механически, выключаются при reduced-motion.
- Город будущего: читаемый силуэт + параллакс + летающие машины + ≥3 типа случайных событий
  (такси/полиция/дирижабль), всё оригинальное и дешёвое.
